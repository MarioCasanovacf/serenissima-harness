A society of agents

Serenissima
Harness

A file-based coordination substrate that lets several AI agents work on one real codebase in parallel — without stepping on each other, without approving their own work, and leaving an auditable trail behind every decision.

01The load-bearing layer

No database. No external APIs. No framework to buy into. Plain files, nine stdlib-only Python CLIs, and a set of rules that are programs for agents to really work — an agent can’t claim a blocked task or edit a file someone else is holding because the command refuses it, not because it agreed to behave.

It was built for Claude Code, but nothing in the contract is Claude-specific. Any agent — or human — that can run a CLI can join the board. What it is not: an agent framework, a runtime, or a prompt library. It is the boring layer underneath that decides who does what, who is allowed to say it is done, and how you would ever reconstruct what happened.

02Why it exists, and who does the work

Two failure modes show up the moment you run more than one coding agent at once: unmanaged parallelism — no shared index, no ownership, so agents duplicate work and overwrite each other — and the giant cascade, one long A→B→C chain where every hop loses context and one stalled link freezes the pipeline. The resolution is a simple idea: delegation is a property of the task graph. A cascade exists exactly where one task consumes another’s artifact and nowhere else; everything else runs in parallel under leased claims and locks.

The name comes from how I pictured it: the whole city of Venice in the Renaissance working toward your goal — a society of agents coordinated by institutions, not by a single brain. Venice lasted roughly eleven centuries because its procedures governed it. The intelligence lives in the agents; the substrate that coordinates them is deliberately dumb, deterministic, and auditable.

Fig. 1A human states a goal; the coordinator decomposes it and dispatches a planner, a pool of workers, and verifiers, all coordinating through one shared board on disk.

03Three layers on plain files

The design follows a three-layer split — Control (the rules and personas, written as natural-language documents an agent reads), Agency (the CLI tools that enforce those rules), and Runtime (the file substrate that holds all shared state). Nothing lives in a hidden service; every layer is a file you can open, diff, and version.

04The rules, and what enforces them

Each rule is backed by a mechanism, not a good intention. The task lifecycle is a small state machine, and the one rule that matters most — nobody approves their own work — is enforced at the transition into done.

RuleMechanism
No claiming a blocked taskblackboard.py claim refuses any task whose depends_on aren’t done
No editing a file someone else holdsTTL write-locks and a hook that blocks the edit before it lands
No agent can freeze the swarmClaims carry leases, locks carry TTLs; a crashed agent’s work expires back to the pool
No one approves their own work--status done is refused unless the task was reviewed and the actor differs from the producer of record
No silent self-modificationThe harness changes itself only through an audit loop with a human gate; git is the undo button

05The evolution ledger

The harness rewrites its own natural-language spec through an audit loop: logs, then evidence-cited proposals, adversarial verification, a human gate, and a generation bump. 4 generations so far, with 25 accepted mutations — and the audits have caught themselves, once rejecting a proposal for a wrong evidence count and once catching the coordinator’s own arithmetic error.

Fig. 3The evolution loop. Frictions become audit inputs; audits become generations. Every change is gated by a human and reversible through git.
Limits

The newest decision was a rejection. Two external ideas were evaluated for generation 5 — a Git-for-agent-runs checkpointer and a self-correcting trace→fix→regression loop — and both were declined: the failure modes they solve have not occurred in this harness’s own trajectories, so they were logged with falsifiable revisit triggers rather than adopted on theory. A well-argued reject beats an enthusiastic adopt.

The harness helped build itself, evolved itself across 4 generations under low supervision, and shipped two real projects under its own rules. mdtoc (a Markdown table-of-contents generator, Python) and cronsplain (a cron-expression explainer, zero-dependency Node.js) were each planned, built, and tested on the board — every task with producer ≠ approver, every verdict adversarially replayed, each winning implementation chosen by a tournament and promoted byte-for-byte.
