Universal Agent Harness

A file-based coordination substrate that lets several AI agents work on one real codebase in parallel — without stepping on each other, without approving their own work, and leaving an auditable trail behind every decision.

No database. No external APIs. No framework to buy into. Plain files, nine stdlib-only Python CLIs, and a set of rules that are programs for agents to really work — an agent can't claim a blocked task or edit a file someone else is holding because the command refuses it and not because it agreed to behave.

It was built for Claude Code, but nothing in the contract is Claude-specific. Any agent — or human — that can run a CLI can join the board. The enforcement hooks are Claude-side convenience; the coordination rules are universal.

What this is not

Let me get this out of the way first, because it's the part most likely to save you time.

  • It is not an agent framework. It doesn't give you agents, prompts, memory, or a runtime. You bring those.
  • It is not a replacement for Claude Code. It sits on top of Claude Code (or any CLI-capable agent) and gives a group of them a shared task board, file locking, and a hand-off protocol.
  • It is not a headless multi-LLM swarm. That was the original ambition — several different models running unattended against one board. The file substrate genuinely supports it, and there's a documented prompt-bridge for a second engine (Gemini), but a fully autonomous cross-model swarm was left out of proven scope: I have not exercised it end-to-end enough to claim it works, so I don't. The mechanics are there, so in the future it will maybe be changed.

What it is: the boring, load-bearing layer underneath multi-agent work that actually gets the job done — this is the part that decides who does what, who's allowed to say it's done, and how you'd ever reconstruct what happened.

Why it exists

Two failure modes show up the moment you run more than one coding agent at once:

  • Unmanaged parallelism. No shared task index, no ownership, no dependency gates. Agents duplicate work, overwrite each other, and nobody actually delegates.
  • The giant cascade. The tempting fix — one long chain A→B→C→…→N. It trades chaos for fragility: every hop loses context, and one stalled link freezes the whole pipeline.

The resolution I propose is a very simple idea: delegation is a property of the task graph. It is not necessarily how the org chart should be established within an agentic framework. That's why work is a dependency-DAG on a shared blackboard. A cascade exists exactly where one task literally consumes another's artifact and nowhere else. Everything else runs in parallel under leased claims and TTL write-locks, so nothing collides and nothing can stall the board forever.

The rules, and what enforces them

Each rule is backed by a mechanism, not a good intention:

RuleMechanism
No claiming a blocked taskblackboard.py claim refuses any task whose depends_on aren't done (the cascade gate)
No editing a file someone else holdsTTL write-locks + a PreToolUse hook that blocks the edit before it lands
No agent can freeze the swarmClaims carry leases, locks carry TTLs — a crashed or looping agent's work expires back to the pool
No one approves their own work--status done is refused unless the task went through review and the actor differs from the producer of record
No self-whitelistingSession-holder registration is coordinator-gated by ambient identity
No unsupervised escalationHuman gates — git push, deletions, first webhook, mutating an agent's constitution — always require an explicit human yes
No silent self-modificationThe harness changes itself only through an audit loop: logs → evidence-cited proposals → adversarial verification → gated apply → generation bump (git is the undo button)

Proof it works (the track record)

I didn't want to write a README full of adjectives, so here's the part you can check with grep. This harness helped build itself, and with low supervision evolved itself across four generations, and shipped two real projects under its own rules — every task with producer ≠ approver, every verdict adversarially replayed. Across the whole record, 55 tasks passed through the producer ≠ approver hand-off.

  • mdtoc (Python) — a Markdown table-of-contents generator, and the harness's first tournament: three independent slugger implementations written with deliberately different methods, judged on real-input fidelity, and the winner promoted verbatim. 62 tests (63 after a later fix).
  • cronsplain (Node.js, zero dependencies) — a cron-expression explainer and next-occurrence engine. Planned with a formal Unknowns pass (a four-quadrant blindspot interview run before the DAG was published), a second tournament (brute-force vs. field-cascade candidates cross-diffed over 450 inputs), and 93 tests.
  • Two tournaments total — one per project — each promoting a byte-identical winner rather than a paraphrase of it.
  • Four generations of self-audits with falsifiable proposals, and the audits caught themselves: one audit was rejected for a wrong evidence count and had to re-diagnose; a verifier once caught the coordinator's own arithmetic error. Across those generations the evolution loop produced 25 accepted mutations (P-001–P-026, with P-023 still in the gen-5 backlog).
  • A swarm smoke test: two concurrent OS processes racing one board — zero double-claims, cross-identity locks held.

The full evidence trail (task files, verdicts, event logs, audits) ships in the repo. Nothing here is claimed that the logs can't confirm.

Quickstart


# 1. Get it into your project. One command, and it never overwrites your files:
python3 .harness/bin/migracion_proyecto.py /path/to/your/repo --dry-run   # preview
python3 .harness/bin/migracion_proyecto.py /path/to/your/repo             # transplant

# 2. Open Claude Code IN your repo (the hooks live where the session is rooted) and say:
#    "Read ORCHESTRATION.md and USAGE.md. My goal: <describe it>.
#     Planner: decompose it onto the board and dispatch."

# 3. Watch the board:
python3 .harness/bin/blackboard.py status

Requirements: Python ≥ 3.9 (stdlib only), git, and any Claude Code plan. The coordinator is a role, not a model — it runs the same on Opus or Sonnet.

Design lineage — what inspired this, and why

This isn't a from-scratch invention; it's a synthesis. Each of these shaped a specific part of the harness:

  • The blackboard-architecture tradition (Erman et al., Hearsay-II, 1980) — why: it's the oldest good answer to "how do independent workers share state without a central brain." The shared blackboard.json task board is a direct descendant.
  • Natural-Language Agent Harnesses & the CAR decomposition — arXiv:2603.25723 — why: it gave me the three-layer split the whole repo is organized around — Control (rules/personas), Agency (the CLI tools), Runtime (the file substrate) — and the idea that an agent's "constitution" is a plain document you can version and mutate.
  • Nenad Tomašev — "society of agents" & the serial-stagnation critique — Google DeepMind: The Podcast, "When millions of AI agents meet" — why: this is the argument against the giant cascade. It's why delegation here is a dependency-DAG (parallel where possible, cascade only where an artifact is truly consumed) instead of one long fragile chain.
  • DeepMind's AI Co-Scientist — Google DeepMind blog — why: the tournament/consensus pattern. When a task has more than one reasonable solution, the harness spawns N candidates with deliberately different methods and lets a judge pick — that's how both mdtoc and cronsplain chose their winning implementations.
  • Intelligent AI delegation — arXiv:2602.11865 — why: capability contracts (which agent is trusted with which work) and the producer ≠ approver rule that no one grades their own homework.
  • ReContext — arXiv:2607.02509 — why: evidence replay. Verifiers don't trust notes; they re-run the hand-off commands themselves. recontext.py and the "replay, don't trust" invariant come from here.
  • Agentic harness engineering — arXiv:2604.25850 — why: the general grounding for "improve the harness, not the weights" — the observability pillars and the evolution loop that turns logs into the next generation.
  • Thariq Shihipar's unknowns framework (known/unknown × knowns/unknowns) — the-decoder.com coverage · prompt post — why: this is what lets me place a context-wise questionnaire into the harness for the user before the agents start working. The planner runs a four-quadrant blindspot pass and asks the human the questions only the human can answer, instead of guessing and discovering the mistake halfway through.

Frictions become audit inputs; audits become generations. The tool that coordinates the work is itself work the tool coordinated.
